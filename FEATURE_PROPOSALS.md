### #1 レシート自動分類の学習機能 — **本 Stage で実装**

- **動機**: カテゴリを毎回ポチポチ直すのが手間。過去の修正履歴から「この店／このメモはこのカテゴリ」と学ばせたい。
- **採用アプローチ (Option A / 軽量 few-shot)**:
  - ユーザが `edit-transaction-dialog` でカテゴリを変えたら `category_corrections` テーブルに記録
  - チャット / レシートスキャン時に「直近 N 件の修正例」をプロンプトに few-shot として注入
  - embeddings は使わない（コストと導入複雑度を避け、Gemini のコンテキスト長に投げる方がシンプル）
- **将来の拡張 (Option B)**: 十分にデータが貯まったら embeddings + 類似度検索に切り替え。`category_corrections` テーブルは両方式で再利用可能。
- **スキーマ**: `category_corrections` 新規（本 Stage の SQL を参照）

### #7 月次 AI レポート — **本 Stage で実装**

- **動機**: 月ごとに振り返りをしたいが、分析画面を開きに行く習慣化が難しい。AI から "届く" 形にしたい。
- **配信**: Push 通知 + チャット画面上部にピン留めバナー

- **スキーマ**: `monthly_reports` 新規

## 📋 将来実装候補（未着手）

### #2 予算オーバー予兆通知

- **動機**: 「月末に予算超えて気づく」を防ぎたい。週次のペースから月末予測を出して「このままだと食費がオーバーしそう」と早めに警告。
- **概要**:
  - 毎週月曜朝に cron 起動
  - ユーザごとにカテゴリ別の日割り消費ペースを計算
  - `(現在支出 / 経過日数) × その月の総日数` が予算を超えそうなカテゴリを抽出
  - 該当があれば Push で早期警告（notificationType: `budget_pace_alert`）
- **スキーマ変更**: 不要（既存 `transactions` + `budgets`）
- **実装難度**: 低（cron 1 つ追加 + push 既存インフラ流用）
- **既存 `budget_alert` との違い**: これは「80%/100%到達後」の事後通知。本案は「このペースだと月末に超えそう」という予測通知。

### #3 割り勘 / 精算モード

- **動機**: 共同費を片方が立て替えた時、月末にまとめて精算する。今の「共同」区分は按分計算されていない（ただ共同口座的に記録しているだけ）。
- **概要**:
  - `payments` テーブル新設: `payer_user_id`, `amount`, `split_rule (jsonb)`, `settled_at`
  - 毎回「どちらが払ったか」「按分比率」を記録
  - ダッシュボードに「今月の精算額: れん→あかね ¥X,XXX」を表示
  - 精算実行ボタンで `settled_at` を更新
- **スキーマ変更**: `payments` 新規。`transactions.payer_user_id` カラム追加案もあり
- **実装難度**: 中（UI 含めて丸ごと新ページ必要）
- **懸念**: 既存「共同」運用を壊さないよう、既存データとの互換モードが要る

### #4 買い物メモ連携

- **動機**: 「買い物リスト」を事前に書いておいて、レシートスキャン時に突き合わせて「買い忘れ」を検出する。
- **概要**:
  - `shopping_lists` テーブル: `user_id`, `items (jsonb)`, `created_at`, `completed_at`
  - 買う前にチャット/ UI から「牛乳 / 卵 / パン」と登録
  - レシートスキャン時に AI が突き合わせ → 「卵が入っていません。買い忘れ？」
- **スキーマ変更**: `shopping_lists` 新規
- **実装難度**: 中（UI + Gemini プロンプト拡張）
- **面白さ**: 高。#1 の学習と合わせると「いつもこのスーパーで卵買ってるけど今回ないよ」みたいに言える

### #5 サブスク管理

- **動機**: 固定費テーブルにサブスクも混ざっているが、値上げや「実は使ってない」に気づきにくい。
- **概要**:
  - `fixed_expenses` に `type` カラム追加: `utility / rent / subscription / insurance / other`
  - サブスク種別だけ別画面で一覧 + 直近の使用記録 (手動チェックイン) を並べる
  - 3ヶ月チェックインなしなら「これ解約候補？」と提案
  - 金額変動検知: 前回と違う額で記録されたら「値上げ？」通知
- **スキーマ変更**: `fixed_expenses.type` カラム追加、`subscription_usage` 新規
- **実装難度**: 中（UI 増 + 金額変動検知ロジック）

### #6 キャラクター進化

- **動機**: 既にキャラ着せ替え機能がある（Hachiware）。節約達成度に応じてアクセサリや新モーションが解放されるとモチベが続く。
- **概要**:
  - `character_unlocks` テーブル: `user_id`, `character_id`, `unlock_key`, `unlocked_at`
  - 解放条件: 「3ヶ月連続で食費予算内」「貯金目標 1 つ達成」など
  - 進化形アセットは `lib/characters/` 側に `evolution: {...}` で生やす
- **スキーマ変更**: `character_unlocks` 新規
- **実装難度**: 中〜高（アセットデザインが別途必要）
- **依存**: 既存 Hachiware 実装が安定してから
